Documentation

Install the package, learn the rules, and work inside the frozen 0.1 surface.

VeloraCSS ships as compiled CSS, optional runtime helpers, and a manifest-backed API. These docs cover the public rules that should not drift between releases.

Install

npm install veloracss
import "veloracss/css";
import { initVelora } from "veloracss/js";

Rules

Public rules

Prefixes are strict

Selectors use vel-, tokens use --vel-, and runtime hooks use explicit data-vel-* attributes.

Geometry stays sharper

Pill-heavy chrome is not the default. Rounded pills are opt-in rather than the baseline framework look.

The manifest is authority

The full public selector and token surface is generated into dist/velora.manifest.json during build.

Quick start

Get a real page on screen in three moves

1

Install the package

Bring in the compiled CSS first. Add the runtime only if you need overlays or simple open and close behavior.

npm install veloracss
import "veloracss/css";
2

Start with layout, then add components

Use the shell, stacks, grids, and readable widths to frame the page before dropping in cards, buttons, or forms.

vel-shell vel-stack-lg vel-grid-two vel-max-w-copy
3

Opt into runtime helpers only when needed

Menus, modal shells, and toast stacks can stay declarative by using the shipped runtime hooks instead of custom glue code.

data-vel-toggle data-vel-open data-vel-close

  </section>
</main>

How to think about it

Velora is not utility soup

  • Use layout primitives first so the page structure is obvious.
  • Use typography roles to establish hierarchy before adding visual effects.
  • Use starter components where they save time, not as mandatory wrappers around everything.
  • Keep custom site chrome outside the framework API if it is product-specific.

Foundation

Layout and composition

Start with the page frame and spacing rhythm, then layer rows, grids, and alignment helpers on top.

vel-shell vel-stack-* vel-grid-* vel-justify-*

Think container, rhythm, layout, and alignment before components.

Tokens and utilities

Spacing, sizing, surfaces, effects

These utilities control spacing, readable widths, surface tone, border treatment, and elevation.

vel-p* vel-gap-* vel-max-w-* vel-bg-* vel-shadow-*

Use these to shape density and hierarchy without custom one-off CSS.

Typography

Readable roles and tone

Type roles cover large hero headings, section titles, body copy, supportive tone, and monospace accents.

vel-display vel-headline vel-body vel-text-* vel-font-*

The goal is consistent roles, not a giant bag of random font overrides.

Core UI

Forms, cards, buttons, alerts

These are the starter controls for real product screens: actions, forms, validation copy, and status messaging.

vel-button* vel-form vel-input* vel-alert*

The baseline stays sharper and more restrained than pill-heavy UI kits.

Navigation and flow

Shell orientation and progress

Use these pieces to orient the page, move between views, and show staged progress without a heavy runtime layer.

vel-navbar* vel-breadcrumb* vel-tabs vel-steps

They are meant for docs shells, dashboards, and setup flows.

Actions and overlays

Focused interaction surfaces

This group covers toolbars, grouped actions, menus, modal shells, toast feedback, and the small runtime hooks that support them.

vel-action-* vel-dropdown* vel-modal* data-vel-toggle

The JS surface stays narrow: open, close, and toggle only.

What a good Velora form should do

Guidance

What each pattern is for

  • Use vel-navbar* for the top-level shell and high-value routes.
  • Use vel-breadcrumb* when the user needs location context inside deeper content.
  • Use tabs for peer views, not for primary site architecture.
  • Use pagination when users move through ordered sets rather than jumping between product areas.
vel-navbar* vel-breadcrumb* vel-tabs vel-pagination

Guidance

Where the helper layer should stop

  • Use action bars for clustered task controls and dense page tools.
  • Use dropdowns for short action lists, not deep navigation trees.
  • Use toasts for non-blocking confirmation or status changes.
  • Reach for your app layer when interaction needs outside-click logic, viewport-aware positioning, or complex state.
vel-action-bar vel-icon-button vel-dropdown* vel-toast*

Runtime

When to use the helper layer

Use it for

  • Opening and closing dropdown menus.
  • Showing and dismissing modal shells.
  • Toggling toast stacks or lightweight status surfaces.

Do not use it for

  • Complex application state or data fetching.
  • Viewport-aware positioning logic beyond the shipped basics.
  • Rebuilding component behavior that should live in your app layer.
